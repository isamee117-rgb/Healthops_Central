Database Architecture — 3 Approaches

How should form configuration be stored? This shapes everything else.

A

4 Fully Normalized Tables

Every layer gets its own table with proper foreign keys.

form_groups (id, name, context, description, sort_order)
forms (id, form_group_id, name, description, is_active, sort_order)
form_sections (id, form_id, title, sort_order)
form_components (id, form_section_id, type, label, sort_order, config JSON)

Pros

  • Cleanest Laravel models
  • Each level independently editable
  • Proper relationships & eager loading
  • Queryable by component type

Cons

  • 4 migrations, 4 models, 4 controllers
  • More API endpoints needed

⭐ Recommended — cleanest, most maintainable, follows your codebase pattern

B

2 Tables — Full JSON Schema

Form stores its entire structure (sections + components) as one JSON blob.

form_groups (id, name, context, sort_order)
forms (id, form_group_id, name, is_active, schema JSON)

// schema = { sections: [{ title, components: [...] }] }

Pros

  • Only 2 tables/models
  • Fewest API endpoints
  • Easy to version/duplicate forms

Cons

  • Can't query by component type
  • Large JSON blobs get unwieldy
  • Harder to validate individual components
C

3 Tables — Sections Normalized, Components as JSON

Upgrades the existing pattern: form_sections gets a foreign key to forms, components stay as JSON array in fields.

form_groups (id, name, context, sort_order)
forms (id, form_group_id, name, is_active, sort_order)
form_sections (id, form_id, title, sort_order, fields JSON)

// Mirrors existing opd_form_sections pattern

Pros

  • Close to existing codebase pattern
  • Sections independently manageable
  • Simpler than 4 tables

Cons

  • Components not individually queryable
  • JSON fields array grows large for complex forms
